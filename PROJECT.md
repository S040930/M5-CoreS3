# PROJECT.md

## Core Scope
M5Stack CoreS3 双模固件：AirPlay 1 接收播放 + 最小稳定语音助手 + 本地优先环境提醒。当前语音链路已收口为本地 ESP-SR AFE/WakeNet(`Hi ESP`)/VAD 单轮唤醒切段，随后通过 DashScope OpenAI-compatible Chat Completions 调用 `qwen3.5-omni-flash` 获取一次性回复；请求固定为非思考模式并同时要求 `text/audio` 输出，以便返回 `delta.audio.data`。每轮 assistant 播放结束后立即回到未 armed 的 wake-wait 状态，必须重新说唤醒词；旧的 follow-up window、多轮上下文续写、模型工具调用 followup 链已从主路径移除。ENV.3 传感器子系统本地读取 `SHT30 + QMP6988`，按季节和阈值先做本地判断，只有越界时才通过独立 one-shot `realtime_voice_speak_text()` 播报短提醒；气压仅记录诊断。语音回复继续复用共享 `audio_output` 外部所有权/写出路径，以保持与 AirPlay 一致的时钟、限幅和底噪行为，稳定性优先于复杂能力。

## Recent Changes
- 2026-05-07: **voice_fe_v2 WDT starvation guard**: `voice_frontend_v2` now performs a cooperative `vTaskDelay(1)` on each successful loop iteration (after mic read + AFE feed/fetch), ensuring CPU0 idle task gets scheduled and task WDT remains fed under sustained frontend activity.
- 2026-05-07: **AirPlay resume trigger after voice owner release**: `voice_controller` now calls the configured AirPlay refresh callback when voice playback stops and when AirPlay preempts an active voice reply. This ensures `airplay_service_refresh_playback()` retries pipeline startup immediately after `voice_v2` releases external speaker ownership, avoiding stuck playback when AirPlay was previously blocked by `external owner`.
- 2026-05-07: **Voice speaker-format contract / single boundary adapter**: `voice_request` now parses first-packet audio metadata, including optional WAV headers, so the active turn records the cloud stream's actual sample rate / channels / bit depth once instead of inferring it in several modules. `voice_playout` keeps the current stream format as explicit state, `voice_speaker` sizes its write buffers from that live format, `voice_dsp` builds the playout resampler only from the stream rate, and `voice_reference` now treats the speaker output rate as the reference source rate. The result is a single format-adaptation boundary with first-packet format logs rather than repeated hidden assumptions.
- 2026-05-07: **Minimal stable voice refactor**: `realtime_voice` 主链路主动裁剪为“本地唤醒 -> 单轮录音 -> 一次性 HTTP 请求 -> 流式音频播报 -> 回到未 armed”。`voice_frontend` 收口为单 worker 方向，不再保留复杂 pause/resume barrier 之外的双任务 fetch 状态机；`voice_request` 移除 `tool_calls` 解析、tool dispatch 和 followup request 递归；`realtime_voice_speak_text()` 明确降为独立系统播报入口，不再参与用户会话态。`docs/voice-interaction.md` 同步改写为最小稳定语音路径。
- 2026-05-07: **Voice reply playback stabilization on dual-core path**: `voice_playback_task` now runs on CPU1 in dual-core builds and drains assistant audio in bounded cooperative bursts instead of monopolizing CPU0. One-shot reply playout raises startup/rebuffer watermarks, treats underrun as `silence + rebuffer` rather than replaying stale audio, and adds slow-write diagnostics around the shared `audio_output` path. This directly targets the observed `voice_playback -> voice_playout_drain -> voice_rs_process_mono -> resampleProcessInterleaved` WDT chain and the matching field symptoms of slow, choppy, noisy speaker output.
- 2026-05-06: **Voice playback path unification**: realtime voice assistant replies now use the shared `audio_output` external-owner path instead of direct codec open/write/close calls, removing the voice-only speaker boost and keeping output gain/headroom aligned with AirPlay. `voice_request` also stopped hardcoding `24000` for decoded assistant audio and now uses `CONFIG_VOICE_OUTPUT_SAMPLE_RATE`; voice gap-close timeout was relaxed to reduce premature close on short stalls.
- 2026-05-06: **Voice upload preflight heap-corruption guard**: `voice_rs_ensure_float_bufs()` now grows shared resampler float buffers by allocating replacements before swapping pointers/capacity, with explicit size-overflow and allocation-failure checks so a failed grow cannot leave stale buffer state behind. Codec-facing voice scratch buffers now use internal DMA-capable allocation (`voice_dma_buf_alloc`) instead of PSRAM-first allocation for the mic read buffer and final speaker stereo write buffer. `voice_request_send_audio()` also adds heap integrity checkpoints around WAV, base64, and audio request-body construction, making any remaining pre-upload corruption visible at the first failing stage instead of only surfacing as a TLSF `LoadProhibited` panic.
- 2026-05-06: **Voice frontend pause barrier + playback WDT fix**: `voice_frontend_pause()` is now a cooperative pause barrier: capture/fetch tasks must acknowledge `pause_requested` before the frontend resets AFE queues, reference rings, and runtime accumulators, preventing `afe_bridge_reset()` from racing active `feed()` / `fetch()` during one-shot upload pauses. One-shot upload now aborts if that barrier times out instead of continuing with unsafe frontend state. `voice_playback_task` drains playout in bounded bursts with tick yields so long assistant playback no longer starves CPU0 idle and trips task WDT. Request heap integrity checks are now behind `CONFIG_VOICE_REQUEST_HEAP_CHECK_ENABLE` instead of running in the default production path.
- 2026-05-06: **AirPlay rebuffer escalation for sustained starvation**: actual runtime AirPlay defaults now move to `target_latency_ms=280` and `max_buffer_ms=420` in `config/config.toml` / Kconfig, not just the example file. `audio_output` rebuffer recovery now uses a three-stage policy: low-water entry, high-water resume (`max(target+6, ceil(1.5x target))` plus consecutive-ready streak), then a short cooldown hold to avoid immediate rebuffer thrash. Rebuffer `nack/drop/underrun` logs now compare against the previous stable-play baseline instead of the most recent real-frame snapshot, so field logs expose real starvation pressure instead of repeated `nack+0 drop+0` false calm.
- 2026-05-06: **ENV.III missing-ACK conclusion + AirPlay low-water recovery**: `env_monitor` no longer treats `bus_add_device()` success as a valid `QMP6988` probe; it now requires a real `chip_id == 0x5C` match on `0x70` or fallback `0x56`, and when `0x44/0x56/0x70` all miss ACK it logs an explicit `env probe failed` conclusion and backs off retries to a lower-noise cadence. AirPlay realtime playout now watches sustained low buffer watermarks plus zero-read/NACK bursts, enters a clear `low-water rebuffer` mode instead of looping forever in concealment, and resumes real playback only after buffered frames recover to the target level. Default AirPlay latency/buffer headroom is raised to `220/320 ms` to better tolerate Wi-Fi jitter.
- 2026-05-06: **Reference-ring gate fix for AirPlay**: `reference_ring` is now treated strictly as live AEC input, not a backlog queue. AirPlay reference tap drops data whenever `voice_frontend` is paused/stopped, and `voice_frontend` pause/resume/stop edges reset the ring so stale reference audio cannot survive an AirPlay gate or oneshot-upload pause. Overflow logging for `reference_ring` is now rate-limited and includes `frontend_running/paused` plus `output_active` context instead of spamming one line per dropped chunk.
- 2026-05-06: **ENV.III I2C NACK diagnostics**: `env_monitor` now logs a one-shot shared-bus discovery snapshot and reports per-device/per-register failures for `SHT30@0x44` and `QMP6988@0x70/0x56`, so `unexpected nack` no longer collapses into a generic `sensor init retry pending`. The log explicitly separates “address ACK exists” from “chip ID / register protocol matches current `QMP6988` driver assumptions”, while also reminding that CoreS3 I2C is shared with board codec control and `LTR553` auto-brightness.
- 2026-05-06: **Voice request heap-corruption hardening**: `voice_request_send_audio()` now owns all per-request temporaries (`wav_data`, base64 buffer, JSON body, SSE line buffer) through a single cleanup path, and `wav_from_pcm()` / `base64_encode_wav()` now validate size arithmetic before allocation/encode. `upload_start` / `upload_cleanup` logs now include `wav_len` / `b64_len` / `body_len` so second-turn follow-up crashes can be tied to a specific request-construction stage instead of surfacing only as `tlsf_free ... block already marked as free`.
- 2026-05-06: **Spoken timer reminders + request-body ownership fix**: `set_timer` 到点后不再只做 UI/log 提示，`realtime_voice` 现在会把提醒文本排队并在语音链路空闲、AirPlay 未占用输出时自动调用 `realtime_voice_speak_text()` 播报；忙时 reminder 会 deferred，资源恢复后补播一次。与此同时，`voice_request` 里由 `cJSON_PrintUnformatted()` 生成的 request/followup body 改为使用 `free()` 释放，修复 second-turn follow-up 上传在 `voice_buf_free()` 路径踩坏 heap 的崩溃。
- 2026-05-06: **Hi ESP follow-up session window**: 该方案曾把本地 WakeNet 唤醒后的 armed 状态扩展为“限时免唤醒连续对话窗口”。此语义已在 2026-05-07 的最小稳定语音重构中从主路径移除，保留该记录仅用于历史追踪。
