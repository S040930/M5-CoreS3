
# 语音功能重做计划

## 一、现状分析

### 1.1 项目概况
这是一个基于ESP32的AirPlay和语音助手项目，运行在M5 CoreS3设备上。核心功能包括：
- AirPlay音频接收和播放
- 本地WakeNet唤醒词检测
- 单次HTTP请求到DashScope API
- 流式音频回复播放
- 与AirPlay的互斥控制

### 1.2 现有问题
从代码库和PROJECT.md可以看出：
1. **代码复杂度过高**：经过多次修复和补丁，代码变得难以维护
2. **模块耦合严重**：各模块之间职责不清，相互依赖过多
3. **状态管理混乱**：缺乏清晰的状态机管理
4. **内存安全问题**：历史上多次修复内存问题，说明存在系统性问题
5. **日志过于繁杂**：大量诊断日志影响代码可读性
6. **缺乏测试**：没有单元测试和集成测试

### 1.3 现有架构
当前语音功能分布在以下模块：
- `realtime_voice.c/h` - 主编排器
- `voice_session.c/h` - 会话管理
- `voice_request.c/h` - HTTP请求处理
- `voice_playout.c/h` - 音频缓冲
- `voice_playback_task.c/h` - 播放任务
- `voice_speaker.c/h` - 扬声器控制
- `voice_frontend.c/h` - 前端音频处理
- `voice_common.c/h` - 通用工具
- `afe_bridge.c/h` - AFE桥接
- `voice_reference.c/h` - 参考音频
- `voice_dsp.c/h` - DSP处理
- `voice_timers.c/h` - 定时器
- `voice_tools.c/h` - 工具调用

## 二、重写目标和原则

### 2.1 核心目标
1. **保持API兼容性** - 确保现有调用代码无需修改
2. **简化架构** - 清晰的模块划分，职责单一
3. **提高稳定性** - 消除内存问题和死锁
4. **增强可维护性** - 清晰的代码，完善的注释
5. **提高可测试性** - 模块化设计，便于单元测试

### 2.2 设计原则
1. **最小可行设计** - 只保留当前使用的功能
2. **清晰的状态机** - 明确的状态转换
3. **资源所有权明确** - 谁分配谁释放
4. **错误处理健壮** - 完善的错误恢复机制
5. **日志精简实用** - 只保留必要的诊断信息

## 三、新架构设计

### 3.1 模块划分
新架构将采用以下5个核心模块：

```
┌─────────────────────────────────────────────────────┐
│                   voice_controller                   │
│            (主状态机 + 模块协调器)                   │
└───────────┬───────────────────┬─────────────────────┘
            │                   │
┌───────────▼──────────┐ ┌──────▼──────────────┐
│  voice_frontend_v2   │ │   voice_api_client  │
│  (唤醒 + 录音)       │ │  (HTTP请求 + SSE)   │
└──────────────────────┘ └─────────────────────┘
                                    │
                          ┌─────────▼──────────┐
                          │    voice_player    │
                          │   (音频播放)       │
                          └────────────────────┘
```

### 3.2 模块职责

#### 1. voice_controller (主控制器)
- 管理全局状态机
- 协调各子模块
- 处理AirPlay互斥
- 发送UI状态通知
- 实现系统播报接口

#### 2. voice_frontend_v2 (语音前端)
- 初始化AFE/WakeNet
- 唤醒词检测
- 音频采集
- 录音管理和静音检测
- pause/resume控制

#### 3. voice_api_client (API客户端)
- HTTP请求构建和发送
- SSE流解析
- WAV/base64编码
- 音频数据解码
- 错误处理和重试

#### 4. voice_player (音频播放器)
- 与audio_output集成
- 音频缓冲管理
- 预缓冲和播放控制
- 播放状态管理

### 3.3 状态机设计

```
IDLE  ──(唤醒词)──►  RECORDING  ──(录音完成)──►  REQUESTING
  ▲                                                        │
  │                                                        │
  └────────(播放完成/错误)──────────────────────────────────┘
                         │
                         ▼
                      PLAYING
```
